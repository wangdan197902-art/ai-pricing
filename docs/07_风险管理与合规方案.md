## 1. 风险体系总览

### 1.1 风险定义

本项目所指风险包括：可能影响项目目标达成、造成经济损失、声誉损害、法律责任、运营中断的一切不确定事件，包括正面（机会）与负面（威胁）两类。本方案以负面风险防控为主。

### 1.2 风险分类框架（七维）

| 维度 | 说明 | 优先级 |
|------|------|--------|
| 技术风险 | 基础设施、代码、AI API、安全漏洞等 | 中 |
| 内容风险 | 价格准确性、攻略有效性、信息时效性 | **高** |
| 合规风险 | 法律法规、平台政策、商标、广告披露 | **最高** |
| 商业风险 | 商业模式、变现、合作伙伴、竞争 | 高 |
| 运营风险 | 团队、流程、SOP 执行、外部依赖 | 中 |
| 外部风险 | 政策变化、市场环境、不可抗力 | 中 |
| 声誉风险 | 用户信任、品牌形象、负面舆情 | 高 |

### 1.3 风险评估矩阵

采用 **概率 × 影响** 的 5×5 评估模型：

**概率等级**：
- 1 = 极低（<5%）
- 2 = 低（5%-20%）
- 3 = 中（20%-50%）
- 4 = 高（50%-80%）
- 5 = 极高（>80%）

**影响等级**：
- 1 = 可忽略（损失 < $50）
- 2 = 轻微（损失 $50-$500，无声誉影响）
- 3 = 中等（损失 $500-$5000，短期声誉影响）
- 4 = 严重（损失 $5000-$50000，长期声誉损害，监管关注）
- 5 = 灾难性（损失 > $50000，业务中断，法律诉讼）

**风险等级 = 概率 × 影响**：
- 1-4：低风险（绿色，监控即可）
- 5-9：中风险（黄色，需制定应对措施）
- 10-15：高风险（橙色，需主动管控）
- 16-25：极高风险（红色，必须立即处理）

### 1.4 风险应对策略

| 策略 | 含义 | 适用场景 |
|------|------|----------|
| 规避 | 改变计划消除风险 | 法律红线、致命威胁 |
| 减轻 | 降低概率或影响 | 大多数可控风险 |
| 转移 | 保险、外包、合同条款 | 财产损失、第三方责任 |
| 接受 | 主动承担小额风险 | 低概率低影响 |
| 利用 | 抓住正面机会 | 竞争对手失误、政策红利 |

### 1.5 风险管理流程（PDCA 闭环）

```
风险识别 → 风险评估 → 风险应对 → 风险监控 → 风险回顾 → 风险识别（循环）
   ↓            ↓            ↓            ↓            ↓
 风险登记册   评估矩阵    应对计划     指标告警      季度复盘
```

### 2.2 技术风险详解

#### 2.2.1 Cloudflare Pages 服务中断

**场景**：Cloudflare 全球性故障导致站点不可访问。

**应对措施**：
- **预防**：在 `05_部署与运维方案.md` 中已配置 UptimeRobot 监控，5 分钟内告警。
- **应急**：准备 Vercel 作为备选部署平台，本地保留完整构建产物，可在 30 分钟内切换。
- **沟通**：通过状态页（status.example.com）发布故障通告。
- **接受**：Cloudflare SLA 99.99%，年度预期中断 < 53 分钟，可接受。

#### 2.2.2 GitHub 仓库被误删或被封

**场景**：仓库因操作失误或 GitHub 政策原因被删除。

**应对措施**：
- 每日自动备份到 Bitbucket/GitLab（参见 `05_部署与运维方案.md` 备份章节）。
- 本地保留完整 git 镜像。
- 关键分支启用分支保护规则，禁止 force push。
- 使用 GitHub Organization 而非个人账号，避免个人账号被封影响项目。

#### 2.2.3 Hugo 框架升级破坏现有站点

**场景**：Hugo 新版本不兼容现有模板、配置或短代码。

**应对措施**：
- **版本锁定**：`netlify.toml` / Cloudflare 构建配置中指定 Hugo 精确版本（如 `HUGO_VERSION = "0.140.2"`）。
- **升级流程**：
  1. 在独立分支测试新版本（`chore/hugo-upgrade`）
  2. 运行完整构建并人工检查 10 个核心页面
  3. 通过 Preview Deployment 验证
  4. 合并到 main，监控 24 小时
- **回滚预案**：保留上一版本 Docker 镜像，可在 10 分钟内回滚。

#### 2.2.4 AI API 服务中断

**场景**：Claude、GPT-4o、Gemini 任一 API 不可用。

**应对措施**：
- **多供应商策略**：
  - 主：Claude 3.5 Sonnet（对比/攻略生成）
  - 备：GPT-4o（可临时接管对比生成任务）
  - 备：Gemini 1.5 Flash（可临时接管翻译任务）
- **降级模式**：API 不可用时暂停内容生成任务，已生成内容正常发布，待 API 恢复后补生成。
- **任务队列**：GitHub Actions 工作流失败自动重试 3 次，间隔 10 分钟。

#### 2.2.5 价格爬虫被厂商 IP 封禁

**场景**：频繁抓取厂商定价页被反爬机制识别。

**应对措施**：
- **优先使用官方 API**：OpenAI、Anthropic、Google AI 等提供 API 或公开的 pricing 页面 JSON。
- **频率控制**：单 IP 单域名每日抓取 ≤ 1 次。
- **多源比对**：从 AI Directory、TAAFT、Futurepedia、Wayback Machine 多源获取价格，降低单一来源依赖。
- **Claude fallback**：当结构化获取失败时，调用 Claude 解析定价页 HTML（已在 `03_内容获取与生产闭环.md` 实现）。
- **代理池**：仅在必要时使用付费代理（如 Bright Data），MVP 阶段不使用。
- **法律边界**：仅抓取公开页面，遵守 robots.txt，不绕过付费墙。

#### 2.2.6 安全漏洞（XSS/CSRF/注入）

**场景**：静态站点本身风险低，但用户提交（如价格反馈表单）可能引入 XSS。

**应对措施**：
- **静态站点优势**：Hugo 生成纯静态 HTML，无服务端执行，天然免疫 SQL 注入、RCE。
- **表单处理**：用户反馈通过 Formspree/BotDuck 第三方服务处理，不在本站存储或渲染。
- **CSP 头**：Cloudflare `_headers` 配置严格 CSP：
  ```
  Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline' https://cdn.example.com; ...
  ```
- **依赖扫描**：GitHub Dependabot 自动监控依赖漏洞。

### 3.2 价格准确性保障机制

#### 3.2.1 多层校验体系

```
Layer 1: 自动价格抓取（每日）
  ↓ 异常检测（与昨日偏差 > 30% 标记）
Layer 2: Claude 解析校验（HTML 解析失败时）
  ↓ 与多源数据比对（AI Directory / TAAFT / Wayback）
Layer 3: AI 内容审核（audit_content.py）
  ↓ 价格与官网、清单、历史一致性校验
Layer 4: 人工抽检（每周 10 个工具）
  ↓ 用户反馈通道（每页"价格反馈"按钮）
Layer 5: 用户反馈闭环（48h 内处理）
```

#### 3.2.2 价格时效性标准

| 更新频率 | 标签 | 说明 |
|----------|------|------|
| ≤ 24h | "实时" | 当日抓取 |
| ≤ 72h | "近期" | 3 天内 |
| ≤ 7d | "本周" | 一周内 |
| > 7d | "待核实" | 黄色警告标记，触发优先重抓 |
| > 30d | "可能过时" | 红色警告，从首页/榜单移除 |

#### 3.2.3 价格偏差自动检测脚本（伪代码）

```python
# scripts/detect_price_anomaly.py
import json
from datetime import datetime, timedelta
from pathlib import Path

def detect_anomalies(history_file: str, threshold_pct: float = 0.30):
    """检测价格异常波动。
    
    Args:
        history_file: 价格历史 JSON 文件路径
        threshold_pct: 偏差阈值（默认 30%）
    
    Returns:
        list: 异常记录列表
    """
    with open(history_file) as f:
        history = json.load(f)
    
    anomalies = []
    for tool_id, records in history.items():
        if len(records) < 2:
            continue
        latest = records[-1]
        previous = records[-2]
        
        for plan_name, latest_price in latest["prices"].items():
            if plan_name not in previous["prices"]:
                continue
            prev_price = previous["prices"][plan_name]
            if prev_price == 0:
                continue
            change = abs(latest_price - prev_price) / prev_price
            if change > threshold_pct:
                anomalies.append({
                    "tool_id": tool_id,
                    "plan": plan_name,
                    "previous": prev_price,
                    "latest": latest_price,
                    "change_pct": round(change * 100, 2),
                    "date": latest["date"],
                    "severity": "high" if change > 0.5 else "medium"
                })
    
    return anomalies

def send_anomaly_alert(anomalies: list, webhook_url: str):
    """发送异常告警到 Slack。"""
    import requests
    if not anomalies:
        return
    
    message = f"⚠️ 检测到 {len(anomalies)} 条价格异常：\n"
    for a in anomalies[:10]:  # 最多展示 10 条
        direction = "↑" if a["latest"] > a["previous"] else "↓"
        message += (
            f"• {a['tool_id']} {a['plan']}: "
            f"${a['previous']} → ${a['latest']} "
            f"({direction}{a['change_pct']}%)\n"
        )
    
    requests.post(webhook_url, json={"text": message})

if __name__ == "__main__":
    anomalies = detect_anomalies("data/price_history.json")
    send_anomaly_alert(anomalies, "https://hooks.slack.com/services/XXX")
```

#### 3.2.4 价格更正流程

当发现价格错误时：

1. **立即冻结展示**：在数据文件中将该工具价格标记为 `status: "under_review"`，前端显示"价格待核实"。
2. **重新抓取**：触发 `fetch_prices.py` 单工具重抓。
3. **多源验证**：人工访问官网 + AI Directory + Wayback Machine 三方比对。
4. **更新数据**：修正 `price_history.json`，保留更正记录（不删除历史错误数据）。
5. **公告**：如错误已展示 > 24h，在工具页底部添加"价格更正说明"。
6. **用户通知**：订阅价格提醒的用户邮件告知更正。

## 4. 合规风险（重点）

合规风险是本项目的**最高优先级**，处理不当可能导致：账户封禁、罚款、诉讼、声誉破产。本章节详细列出六大合规领域。

### 4.1 GDPR / CCPA 合规

#### 4.1.1 适用范围

- **GDPR**（欧盟通用数据保护条例）：面向欧盟用户提供服务或跟踪欧盟用户数据时适用。
- **CCPA / CPRA**（加州消费者隐私法）：面向加州居民时适用。
- 本项目虽为静态站点，但通过 Google Analytics、AdSense、Ezoic 等第三方服务收集用户数据（IP、Cookie、行为），均触发合规义务。

#### 4.1.2 数据收集清单

| 数据类型 | 来源 | 用途 | 法律依据 |
|----------|------|------|----------|
| IP 地址 | Cloudflare / GA | 流量统计、安全 | 合法利益 |
| Cookie | AdSense / Ezoic / GA | 广告个性化、分析 | 用户同意 |
| 用户反馈内容 | Formspree 表单 | 改进内容 | 用户主动提交 |
| 邮箱（订阅） | 用户主动输入 | 价格提醒 | 用户同意 |
| 浏览行为 | GA / Hotjar | UX 优化 | 用户同意 |

#### 4.1.3 合规措施

**Cookie 同意横幅**（必须）：
- 使用 Cookiebot 或 Termly 等合规工具
- 默认不设置非必要 Cookie
- 提供"接受全部""仅必要""自定义"三个选项
- 记录用户同意时间和选择

**隐私政策页面**（`/privacy/`）：
- 包含数据收集清单、用途、第三方共享、用户权利、联系方式
- 模板详见附录 13.1

**用户权利响应**：
- 删除请求：30 天内删除用户提交的所有数据
- 数据导出：提供用户数据的 JSON 导出
- 反对处理：停止该用户的个性化广告
- 联系方式：privacy@example.com

**数据传输**：
- 仅使用通过 GDPR 认证的服务商（Cloudflare、Google、Amazon 已认证）
- 不向未认证第三方传输用户数据

#### 4.1.4 GDPR 罚款风险

- **最高罚款**：2000 万欧元或全球年营业额 4%（取高者）
- **本项目实际风险**：极低（无敏感数据、无大量欧盟用户、年营收有限），但仍需合规
- **应对**：购买网络责任险（详见第 10 章）

### 4.2 AdSense 政策合规

#### 4.2.1 关键政策要点

| 政策 | 要求 | 违反后果 |
|------|------|----------|
| 内容政策 | 原创内容、无成人/暴力/仇恨内容 | 账户暂停 |
| 流量来源 | 禁止付费流量、禁止激励点击 | 账户封禁 |
| 点击欺诈 | 严禁任何形式鼓励点击 | 账户封禁 + 没收收入 |
| 版权 | 不展示侵权内容 | 账户封禁 |
| 隐私 | 遵守 GDPR/CCPA | 账户暂停 |
| 广告位置 | 不遮挡内容、不诱导误点 | 广告下线 |
| 禁止品类 | 不在赌博、药品、烟草等页面投放 | 账户封禁 |

#### 4.2.2 本项目特定风险

**风险一：内容被判定为"低质或重复"**
- AI 生成内容可能被 Google 判定为 programmatic spam
- **应对**：
  - 每篇内容必须包含人工策划的"编辑点评"段落（≥ 100 字）
  - AI 生成后人工审核修改，版本控制记录修改痕迹
  - 不发布未经审核的纯 AI 内容
  - 控制发布节奏（每日 ≤ 20 篇），避免被判定为内容农场

**风险二：广告点击率异常**
- 价格对比页广告 CTR 天然较高，可能触发 AdSense 反作弊系统
- **应对**：
  - 广告位与内容明显分隔（≥ 20px）
  - 不使用"点击这里""查看优惠"等诱导文案
  - 监控 CTR，高于 5% 时主动检查并可能下线该页面广告
  - 记录所有广告位截图，备查

**风险三：价格对比页被判定为"thin content"**
- 仅展示价格表格、无实质内容的页面可能被判定为薄内容
- **应对**：
  - 每个对比页必须包含：功能对比、适用人群分析、省钱攻略摘要、FAQ（≥ 5 条）
  - 字数 ≥ 800 字
  - 加入"编辑推荐""用户场景"等独家内容

#### 4.2.3 AdSense 账户封禁应急

1. **立即申诉**：通过 [AdSense 申诉表单](https://support.google.com/adsense/contact/appeal_form) 提交，附上整改证据
2. **切换到 Ezoic**：Ezoic 作为备选广告网络（已纳入变现方案）
3. **增加 affiliate 收入**：临时提升联盟链接曝光，弥补广告收入损失
4. **复盘**：封禁原因分析，整改后 30 天内不再申请 AdSense

### 4.3 FTC 披露合规（联盟链接）

#### 4.3.1 FTC 要求

美国联邦贸易委员会（FTC）要求：当存在物质关系（material connection，如联盟佣金）时，必须清晰、显著地披露。

**核心要求**：
1. 披露必须**显眼**（不能藏在页面底部）
2. 披露必须**易懂**（普通用户能理解）
3. 披露必须在**链接附近**（不要求紧贴，但需在视觉范围内）
4. 不能使用**模糊语言**（"赞助"不够，需明确"我们可能获得佣金"）

#### 4.3.2 本项目披露方案

**页面顶部横幅**（所有含联盟链接页面）：
```html
<div class="affiliate-disclosure" role="note">
  <strong>Disclosure:</strong> Some links on this page are affiliate links. 
  If you click and make a purchase, we may earn a commission at no additional 
  cost to you. This does not influence our ratings or recommendations. 
  <a href="/disclosure/">Learn more</a>
</div>
```

**联盟链接标识**：
- 链接旁添加小图标（如 💰 或 "Affiliate" 标签）
- 鼠标悬停显示 tooltip："This is an affiliate link"

**专属披露页面**（`/disclosure/`）：完整披露所有联盟关系，模板详见附录 13.3

**中文版披露**（多语言站点）：
```html
<div class="affiliate-disclosure" role="note">
  <strong>披露声明：</strong>本页部分链接为联盟链接。若您点击购买，我们可能获得佣金，
  您不会因此产生额外费用。这不影响我们的评分和推荐。
  <a href="/zh/disclosure/">了解更多</a>
</div>
```

#### 4.3.3 FTC 违规风险

- **罚款**：每次违规最高 $50,120（2024 年标准）
- **实际风险**：低（小型站点很少被 FTC 主动调查，但需防范竞品举报）
- **应对**：所有页面强制包含披露，CI 检查无披露的页面不予发布

### 5.2 商业风险详解

#### 5.2.1 AdSense 收入不及预期

**原因**：CPC 低于预期、CTR 低于预期、流量不及预期。

**应对**：
- **多元化**：AdSense 仅占收入 60%，搭配 Ezoic、Amazon、Affiliate
- **A/B 测试**：广告位、格式、密度持续优化
- **多语言**：英语 CPC $2-8，其他语言 $0.5-3，多语言覆盖提升整体 RPM
- **监控**：月度收入复盘，低于预期 30% 时启动调整

#### 5.2.2 联盟计划终止

**原因**：AI 厂商调整联盟政策（如 OpenAI 关闭联盟计划）。

**应对**：
- **多厂商联盟**：接入 10+ 厂商联盟，单厂商关闭影响 ≤ 15%
- **直接商务合作**：流量起来后与厂商直接谈 CPS 合作
- **替代方案**：联盟终止后转为"中立推荐"，仍可通过 AdSense 变现

#### 5.2.3 竞品抢占市场

**主要竞品**：Futurepedia、There's An AI For That、AI Directory

**差异化策略**：
- **价格对比深度**：竞品多为列表，本站专注价格对比与省钱攻略
- **多语言**：竞品多为英语，本站覆盖 10 语言
- **SEO 长尾**：聚焦 "X pricing" "X vs Y" 等高商业意图长尾词
- **省钱攻略**：独家"如何省 X"系列内容
- **用户社区**：用户价格反馈、攻略分享

#### 5.2.4 流量增长不及预期

**应对**：
- 月度流量复盘，与 `04_SEO流量与运营变现.md` 预测对比
- 流量 < 预期 50% 时启动应急：
  1. 加大内容产出（每周 +5 篇攻略）
  2. 多渠道引流（Reddit、Twitter、HN）
  3. 优化现有页面（标题、meta、内容深度）
  4. 付费测试（少量 Google Ads 测关键词）

### 8.4 监控脚本示例

```python
# scripts/monitoring/daily_check.py
"""每日合规性检查脚本。
运行方式：python3 scripts/monitoring/daily_check.py
通过 GitHub Actions 每日 09:00 UTC 执行。
"""
import json
import os
from datetime import datetime, timedelta
from pathlib import Path

def check_disclosure_coverage():
    """检查含联盟链接的页面是否都有披露。"""
    content_dir = Path("content")
    issues = []
    
    for md_file in content_dir.rglob("*.md"):
        content = md_file.read_text()
        if "affiliate" in content.lower() or "go.example.com" in content:
            if "affiliate-disclosure" not in content.lower():
                issues.append(str(md_file))
    
    return {
        "check": "disclosure_coverage",
        "total": len(list(content_dir.rglob("*.md"))),
        "issues": issues,
        "status": "PASS" if not issues else "FAIL"
    }

def check_price_freshness():
    """检查价格数据新鲜度。"""
    history_file = Path("data/price_history.json")
    if not history_file.exists():
        return {"check": "price_freshness", "status": "FAIL", "reason": "no data"}
    
    with open(history_file) as f:
        history = json.load(f)
    
    cutoff = datetime.now() - timedelta(days=7)
    stale_count = 0
    total = 0
    
    for tool_id, records in history.items():
        if not records:
            continue
        total += 1
        last_date = datetime.fromisoformat(records[-1]["date"])
        if last_date < cutoff:
            stale_count += 1
    
    stale_pct = stale_count / total if total else 1
    return {
        "check": "price_freshness",
        "total": total,
        "stale": stale_count,
        "stale_pct": round(stale_pct * 100, 2),
        "status": "PASS" if stale_pct < 0.1 else "FAIL"
    }

def main():
    results = [
        check_disclosure_coverage(),
        check_price_freshness(),
    ]
    
    report = {
        "date": datetime.now().isoformat(),
        "checks": results,
        "overall": "PASS" if all(r["status"] == "PASS" for r in results) else "FAIL"
    }
    
    # 保存报告
    report_dir = Path("reports/compliance")
    report_dir.mkdir(parents=True, exist_ok=True)
    report_file = report_dir / f"check_{datetime.now().strftime('%Y%m%d')}.json"
    with open(report_file, "w") as f:
        json.dump(report, f, indent=2, ensure_ascii=False)
    
    # 告警
    if report["overall"] == "FAIL":
        import requests
        message = f"⚠️ 每日合规检查失败：\n"
        for r in results:
            if r["status"] == "FAIL":
                message += f"• {r['check']}\n"
        webhook = os.environ.get("SLACK_WEBHOOK")
        if webhook:
            requests.post(webhook, json={"text": message})
    
    print(json.dumps(report, indent=2, ensure_ascii=False))

if __name__ == "__main__":
    main()
```

### 12.3 风险登记册维护

```python
# scripts/risk_management/update_register.py
"""更新风险登记册。
每次季度回顾后运行，记录风险变化。
"""
import json
from datetime import datetime
from pathlib import Path

def update_risk(risk_id: str, updates: dict, register_file: str = "data/risk_register.json"):
    """更新单个风险条目。
    
    Args:
        risk_id: 风险 ID（如 R-001）
        updates: 更新字段字典
        register_file: 登记册文件路径
    """
    with open(register_file) as f:
        register = json.load(f)
    
    for risk in register["risks"]:
        if risk["id"] == risk_id:
            # 记录历史
            if "history" not in risk:
                risk["history"] = []
            risk["history"].append({
                "date": datetime.now().isoformat(),
                "previous_probability": risk["probability"],
                "previous_impact": risk["impact"],
                "previous_level": risk["level"]
            })
            # 应用更新
            risk.update(updates)
            risk["last_updated"] = datetime.now().isoformat()
            break
    
    with open(register_file, "w") as f:
        json.dump(register, f, indent=2, ensure_ascii=False)

def add_risk(risk_data: dict, register_file: str = "data/risk_register.json"):
    """新增风险条目。"""
    with open(register_file) as f:
        register = json.load(f)
    
    # 生成新 ID
    existing_ids = [r["id"] for r in register["risks"]]
    max_num = max(int(rid.split("-")[1]) for rid in existing_ids if rid.startswith("R-"))
    new_id = f"R-{max_num + 1:03d}"
    
    risk_data["id"] = new_id
    risk_data["created_at"] = datetime.now().isoformat()
    risk_data["last_updated"] = datetime.now().isoformat()
    register["risks"].append(risk_data)
    
    with open(register_file, "w") as f:
        json.dump(register, f, indent=2, ensure_ascii=False)
    
    return new_id
```
